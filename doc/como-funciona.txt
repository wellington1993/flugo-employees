Como o app funciona - de ponta a ponta
Wellington - Desafio Flugo

Escrevi esse texto pra conseguir explicar o funcionamento do sistema
numa entrevista sem precisar abrir o codigo. Vou do mais simples ao mais
tecnico.


O QUE O APP FAZ
---------------

E um sistema de cadastro de colaboradores (funcionarios). O usuario pode:
- Ver a lista de todos os colaboradores cadastrados
- Cadastrar um novo colaborador (nome, email, departamento, status)

E isso. E simples. O desafio nao era o que o sistema faz, era como
eu fiz e que decisoes tecnicas tomei.


STACK TECNICA
-------------

React 19
  Biblioteca JavaScript pra construir interfaces. A ideia central do React
  e dividir a tela em componentes reutilizaveis. Cada componente gerencia
  seu proprio estado e se atualiza automaticamente quando o estado muda.

TypeScript
  JavaScript com tipagem. Em vez de deixar uma variavel ser qualquer coisa,
  voce declara que ela e um "Staff" com campos especificos. O compilador
  avisa antes de rodar se voce estiver usando o tipo errado. Previne uma
  classe inteira de bugs.

Material UI (MUI)
  Biblioteca de componentes visuais seguindo o design system do Google
  (Material Design). Em vez de construir botao, tabela, input do zero,
  importo componentes prontos e customizo com tema.

Firebase / Firestore
  Banco de dados da Google na nuvem. Nao precisa de servidor proprio.
  O front-end se conecta direto no banco usando o SDK do Firebase.
  O Firestore especificamente e um banco NoSQL orientado a documentos,
  parecido com MongoDB.

TanStack Query (React Query)
  Gerenciador de estado de servidor. Cuida do cache, loading, error,
  retry automatico. Sem ele teria que gerenciar tudo isso manualmente.

React Hook Form + Zod
  Hook Form gerencia o estado do formulario (valores, erros, submissao).
  Zod define as regras de validacao em formato de schema. Os dois juntos
  eliminam muito codigo de validacao manual.

Vite
  Ferramenta de build. Compila o TypeScript/JSX, agrupa os arquivos,
  e serve o projeto em desenvolvimento com hot reload instantaneo.

Vercel
  Plataforma de hospedagem. Conecta ao GitHub e faz deploy automatico
  a cada push na main.


FLUXO DE CRIACAO DE COLABORADOR
--------------------------------

1. Usuario clica em "Novo Colaborador"
   O React Router navega pra /colaboradores/novo sem recarregar a pagina
   (isso e o que diferencia SPA de site tradicional).

2. Formulario aparece no passo 1 (nome e email)
   O React Hook Form registra cada campo. O Zod valida em tempo real.
   Se o email ja esta cadastrado, mostra erro imediatamente antes de
   avancar pro passo 2.
   O formulario salva rascunho no localStorage a cada mudanca. Se o
   usuario fechar por acidente, os dados ainda estao la quando voltar.

3. Usuario avanca pro passo 2 (departamento e status)
   O botao "Avancar" so funciona se o passo 1 estiver valido.

4. Usuario clica em "Salvar"
   O React Query dispara a mutacao.
   
   Update otimista: antes de mandar pro Firebase, o item ja aparece na
   lista com a indicacao "Pendente". Isso e o update otimista - a UI
   responde imediatamente sem esperar o servidor.
   
   Requisicao pro Firebase: o servico staffs.ts tenta salvar no Firestore.
   
   Se funcionar: o item na lista perde a indicacao "Pendente".
   
   Se der timeout ou erro: o item fica salvo no localStorage com
   _pendingSync:true e aparece como "Pendente" na lista ate sincronizar.
   O React Query cancela o update otimista (remove o item fake) e adiciona
   o item real do localStorage.

5. Sincronizacao em background
   O hook useSyncPending verifica na inicializacao se tem itens pendentes
   no localStorage. Se tiver, tenta salvar no Firebase em background.
   Se conseguir, remove do localStorage e invalida o cache pra re-buscar
   a lista atualizada.


COMO OS DADOS FICAM ORGANIZADOS NO FIREBASE
--------------------------------------------

Colecao: staffs
  Cada documento tem o email como ID. Isso foi uma escolha deliberada:
  o email ja e unico por natureza, entao usar ele como ID do documento
  garante que nao tenha duplicata sem precisar fazer query pra verificar.
  
  Campos de um documento:
  - name: "Maria Silva"
  - email: "maria@flugo.com" (tambem e o ID)
  - department: "Engenharia"
  - status: "Ativo"
  - createdAt: timestamp

Colecao: app_logs
  Erros do sistema sao salvos aqui. Util pra diagnosticar problemas em
  producao sem precisar ter acesso ao DevTools do usuario.


PERSISTENCIA OFFLINE
--------------------

O Firebase tem uma funcionalidade chamada persistencia local. Quando
habilitada, ele guarda uma copia dos dados no IndexedDB do browser.
Isso significa que:
- O app carrega instantaneamente na segunda visita (dados do cache local)
- O app funciona mesmo sem internet (le do cache)
- Quando a conexao volta, sincroniza automaticamente

O localStorage e usado separadamente so para os itens "pendentes"
(criados sem Firebase). Sao duas camadas de persistencia com propositos
diferentes.


ARQUITETURA EM CAMADAS
----------------------

Services (src/services/)
  Acesso a dados. Nao sabe nada de React. So sabe chamar Firebase
  e localStorage.

Hooks (src/features/staff/hooks.ts)
  Ponte entre services e componentes. Usa React Query pra gerenciar
  cache, loading e error. Os componentes nao chamam services diretamente.

Components (src/components/)
  Interface visual. Usam os hooks pra buscar dados e disparar acoes.
  Nao sabem nada de Firebase ou localStorage.

Essa separacao serve pra: se um dia eu quiser trocar o Firebase por
outra coisa, so mudo os services. Os componentes nem ficam sabendo.


CI/CD
-----

Cada push na branch main dispara o workflow no GitHub Actions:
1. Faz build do projeto (com variaveis de ambiente substituidas por
   placeholders - o Firebase nao precisa conectar pra buildar)
2. Roda os 38 testes unitarios
3. Roda os smoke tests contra a URL de producao ja deployada

O Vercel tambem detecta o push e faz o deploy automaticamente em paralelo.
O smoke test espera um pouco e entao verifica que o deploy funcionou.


O BUG MAIS DIFICIL
------------------

Depois de resolver tudo localmente, em producao o Firebase continuava
dando timeout. Passei horas pensando que era problema de regras de
seguranca, CORS, ou configuracao do Firestore.

O problema real era invisivel: as variaveis de ambiente tinham um
caractere de nova linha (\r\n) no final. O Vercel guardava o valor
"flugo-employees-wts\r\n" e mandava isso pro Firebase que nao
reconhecia o projeto.

Descobri baixando o bundle de producao e inspecionando o valor que
o Firebase estava recebendo. Para corrigir, precisei usar a API REST
do Vercel diretamente porque o CLI do Vercel (e o proprio PowerShell)
tambem adicionavam \r\n no valor ao usar o pipe de terminal.

Isso e um problema especifico de ambiente Windows + PowerShell que
seria transparente em Linux.
